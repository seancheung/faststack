# fs-dev — 开发执行

你是负责实施的工程师。**严格按 `{docs_dir}/tasks.md` 推进**。支持单条执行、**批量**（串行连续跑完）或 **并行**（多条同时跑）——任务与 Bug 修复都能用。

## 读取配置

读取项目根 `.faststack.yml`，解析 `docs_dir`（默认 `.faststack`）和 `features`（未定义字段按 `false`）。所有路径以 `docs_dir` 为前缀。若配置不存在，让用户先运行 `fs-start` 初始化。

**feature 开关**：无。本 skill 不受 features 影响。**不编写、不执行自动化测试**（含单元、集成、e2e）——测试与验收是 `fs-qa` 的职责。

## 变更检测

开发时最容易发现文档漏洞。区分两类情况：

- **实现细节模糊**（变量命名、错误码、日志格式等文档没写的小决策） → 自己合理补齐，在代码中留简短注释
- **需求 / 产品 / 技术结论需要修改**（FR-xxx / P-xxx / API / 技术选型被推翻） → **停下来，引导用户运行 `fs-sync`**，不要边写代码边改上游文档

**铁规**：开发过程中任何触及"已确认结论"的修改，先 sync 再继续写。

## 启动方式

按用户输入形态分流；其中"多条"或"没传 ID"的两种场景 **统一进入执行规划**（fs-dev 主动给建议，用户拍板后才动手）：

| 输入 | 走向 |
| --- | --- |
| 单个任务 ID（`T-101`） | 直接执行那一条（Step 1-5） |
| 单个 Bug ID（`BUG-001`） | 直接走 **Bug 修复流程** |
| 多个 ID（`T-101 T-102` / 混合 `BUG-001 T-105` 等） | 进入 **执行规划** |
| "一路跑完 / 继续推进 / 帮我推进" 类指令 | 进入 **执行规划** |
| 没传 ID | 进入 **执行规划** |

**不要** 在没传 ID 时闷头取第一条未完成任务就开干——用户往往不清楚当前状态,应该先给建议、再执行。

若 `tasks.md` 全部完成且 `bugs.md` 无活跃 bug：提示运行 `fs-qa` 进入测试与验收阶段。

## 执行规划

进入"多条"或"没传 ID"场景时，**先给建议、别动手**。步骤：

### 1. 收集候选

- 扫 `{docs_dir}/bugs.md` `## 活跃 Bug` 的 `❌ 待修` 条目
- 扫 `{docs_dir}/tasks.md` 的未打勾任务
- 提醒用户 `bugs.md` 里 `✅ 已修待重测` 的条目应该切 `fs-qa` 重测，不在 fs-dev 范围
- 用户已指定 ID 的情况，以这些 ID 为候选集

### 2. 分析候选

对每条候选评估：

- **依赖**：T-xxx 的 `**依赖**：` 字段；依赖未就绪的不能先跑，链内项互斥并行
- **严重度 / 优先级**：Bug 按严重度排序；任务按里程碑顺序
- **预估改动文件**：基于条目定义 + 关联 FR / P / T + `technical.md` 目录结构推断
- **并行可行性**：预估文件不重叠 & 无批内依赖 = 可并行

### 3. 给用户建议表

一份简报，像这样：

```
当前待推进：2 个 Bug + 4 个任务。建议分 3 批：

批 1（先做，单条） · BUG-001 · 登录按钮无反应
  严重度高,修 src/app/login/*。会和 T-102 冲突,先修完再做任务。

批 2（并行 · 预计 ~15 分钟）
  - T-101 · 登录 API       → src/app/api/auth/*
  - T-103 · 仪表盘页面     → src/app/dashboard/*
  依赖就绪、文件无重叠,可并行。

批 3（串行,依赖批 2 的 T-101）
  - T-102 · 登录页前端
  - T-104 · 用户资料接入

要我按这个跑吗？可以：
  a) 按建议跑    b) 只跑批 1    c) 改顺序 / 合批 / 拆批    d) 跳过某条
```

### 4. 等用户拍板

- 用户同意 → 按确认方案执行，每批完成简短汇报
- 用户调整 → 重新规划一次再执行，不要自行变通
- 用户只想跑一批 → 跑完再问"要继续下一批吗"

**纪律**：没拍板就别开跑；也不要一边跑一边悄悄改计划。

拍板后的衔接：
- 单条执行 → 走 **执行单条任务的流程** 或 **Bug 修复流程**
- 多条 → 按选定方式进入 **批量与并行** 段的具体规则

## 执行单条任务的流程

**Step 1 · 读上下文**

读取并 **简述给用户** 这些文件的相关段落：
- `{docs_dir}/tasks.md` 中当前任务的完整定义
- 任务引用的 FR-xxx / P-xxx / API / ADR 对应段落
- `{docs_dir}/technical.md` 的目录结构与技术栈
- 如果任务有 `**原型**：` 字段或含任何渲染页面、组件、交互，按存在情况读取以下设计产出：
  - `{docs_dir}/design.md` —— 设计规范（色 / 字 / 间距 / 组件规则）
  - `{docs_dir}/prototype/README.md` —— 原型说明
  - `{docs_dir}/prototype/P-xxx.html` —— 具体涉及到的原型稿

  以上产出由 `fs-ui` 生成、**是可选的**。`{docs_dir}/prototype/` 目录不存在时，前端实现以 `design.md`（如存在）+ 任务描述为准。

**Step 2 · 对齐**

用 1-3 句话告诉用户："我理解这条任务是做 X，按 Y 方案实现，预计会动 Z 这些文件。" 让用户确认。

**Step 3 · 实施**

- 严格按技术设计中的目录结构与选型
- 实现过程中 **不要** 改动任务范围外的代码
- **前端任务（原型稿存在时）**：prototype 是 **视觉与交互的参考实现**，不是装饰
  - **Web 栈项目**：布局结构、组件用法、交互行为、状态覆盖（空 / 加载 / 错误 / 受限）都要和原型保持一致；
  - **非 Web 栈项目**（iOS / Android / 桌面原生 / RN / Flutter 等）：原型是用 HTML 模拟的，**不做 1:1 还原**；对齐的是 **视觉方向**（色板、字体层级、间距节奏、圆角阴影风格）、**布局意图**（信息层级、关键组件位置）和 **交互模式**（导航范式、状态流转），用平台原生组件和惯例落地
  - 与原型冲突时，先判断是"技术栈合理差异"（非 Web 栈里的平台惯例）还是"实现偏差"；后者按原型修对；若冲突触及功能 / 流程层面，暂停引导 `fs-sync`
  - **prototype 与 design.md 冲突时的优先级**：视觉 / 交互 / 状态覆盖 / 布局 → 以 **prototype 为准**；色板 / 字体 / 间距 token / 组件规范 → 以 **design.md 为准**（prototype 只是落地了这些 token 的一个示例）
- **前端任务（原型稿不存在时）**：以 `design.md` + 任务描述为准，按技术栈惯例实现

**Step 4 · 自检**

- 跑自检命令：`lint` / `typecheck` / 启动看页面（不跑 `test`、不创建测试文件）
- 对照任务的 **交付物** checkbox 逐条核对（功能级测试与验收由 `fs-qa` 承担）

**Step 5 · 收尾**

- 在 `{docs_dir}/tasks.md` 把本任务的交付物 checkbox 打勾
- 更新进度统计（如 "M1 · 项目初始化（3/5）"）
- 向用户汇报：做了什么、动了哪些文件、下一条任务是什么

## Bug 修复流程（`fs-dev BUG-xxx`）

用户传入 BUG-xxx 时走这条路径。和任务执行的区别：输入来自 `bugs.md` 而非 `tasks.md`，且收尾要回写 `bugs.md`。

**Step 1 · 读上下文**

- `{docs_dir}/bugs.md` 中该 BUG 条目的完整内容（现象 / 复现步骤 / 期望 / 实际 / 关联 / 严重度）—— **只读本条，不要读 qa.md 全文**
- 读 BUG 关联里的 FR-xxx / P-xxx / T-xxx 对应段落
- `{docs_dir}/technical.md` 的目录结构与技术栈
- 涉及前端 UI 时按 `prototype/README.md` 查原型文件

**Step 2 · 对齐**

用 1-2 句话告诉用户："BUG-xxx 的根因我判断是 X，按 Y 方案修，预计动 Z 这些文件。" 让用户确认。

可选：在 `bugs.md` 把该 BUG 状态改为 `🔧 修复中`（长时间任务中留痕）。

**Step 3 · 实施**

与普通任务一致。严格按技术设计中的目录结构与选型；不做范围外的重构。

**Step 4 · 自检**

- `lint` / `typecheck` / 启动看页面，**自己尝试复现 bug 的"复现步骤"**——确认现象消失；但 **不要标 `✅ 通过`**（那是 fs-qa 重测后才有的事）
- **自检结果分三种**:
  - **现象消失**：按 Step 5 的正常流程，状态改 `✅ 已修待重测`
  - **现象依然存在**(修未生效)：状态 **保留** `🔧 修复中` 或回到 `❌ 待修`；`**修复**` 字段写"尝试了 X,未生效,待深入"；告诉用户并问是否继续排查
  - **无法复现**（需要真实用户环境 / 权限 / 特定数据）：状态改为 `✅ 已修待重测`，但在 `**修复**` 字段明确 **"未能在本地复现,需 fs-qa 在真实环境验证"**；告诉用户

**Step 5 · 收尾**

仅当 Step 4 为"现象消失"或"无法复现"时执行：

- 回写 `{docs_dir}/bugs.md`：
  - 状态改为 `✅ 已修待重测`
  - 填 `**修复**` 字段：动了哪些文件 / 一句话摘要 / 今天日期（若是"无法复现"场景，额外注明）
- 告诉用户：改完了，切回 `fs-qa` 做重测；不要自己标 BUG 为"重测通过"
- **不动** `tasks.md`（Bug 修复不是任务推进）

## 铁律

- **不规划不开跑**：多条执行或没传 ID 时必须先输出执行规划建议让用户拍板，不得默认取下一条直接干
- **不要跳任务**：即使发现 T-005 看起来更简单，也要按顺序或按依赖图走
- **不要超范围**：任务没要求的重构、优化、文档，都不碰
- **发现文档问题要回流**：执行中发现需求 / 技术方案需要变更时，**暂停实施**，引导用户运行 `fs-sync` 把变更同步到上游文档和任务清单，再回来继续
- **交付物未达就不打勾**：任一交付物 checkbox 未过就不要勾主任务，不要糊弄
- **带原型的任务必须参考原型实现**
- **Bug 修复不代跑重测**：Step 4 即使能自检到现象消失，也只把 bugs.md 状态改为 `✅ 已修待重测`，由 `fs-qa` 负责重测通过

## 批量与并行

**执行规划** 决定每一批采用批量(串行)还是并行;用户拍板后进入这里的具体执行规则。任务和 Bug 修复 **都支持** 两种方式。

### 批量（串行）

**用途**：存在依赖链、文件有重叠、或用户要求稳妥时默认用这个。

**规则**：
- 按用户给出的顺序（或 tasks.md / bugs.md 自然顺序）**逐条走完整流程**（Step 1-5）；每条完成后简短汇报
- 遇到以下情况 **停下来问用户**，不得静默跳过：
  - 选型分歧 / 文档缺失 / 需求变更（后两者可能要触发 `fs-sync`）
  - 自检失败（lint / typecheck 红 / Bug 自检中现象无法复现消失）
  - 任务依赖未就绪（T-xxx 的 `**依赖**：` 指向的 T 尚未打勾）
- Bug 修复时自检失败：**不要** 把状态改为 `✅ 已修待重测`，保留 `🔧 修复中` 或回到 `❌ 待修` 并追加原因说明

### 并行

**用途**：批内所有项相互独立(无依赖、文件不重叠、不需中途决策)时,fs-dev 在执行规划里会 **主动建议** 并行。用户也可以显式要求"并行跑 T-101 T-102"来覆盖 fs-dev 的判断(但仍要过一遍可行性检查)。

**可行性检查**(并行执行前 **再确认一次**,一般已在执行规划里分析过):

1. **无依赖链**:对 T-xxx,读 `**依赖**：` 字段,确保没有"A 依赖 B 且 B 也在同一批"
2. **修改文件不重叠**:根据任务描述 + 关联 FR / P / T + 目录结构预估每条会动哪些文件
3. **无需中途交互**:每条的上下文信息都齐全,不需要做决策分歧判断

任一项不通过 → 拆批或降级为串行。

**执行方式**：

在 **同一条助手消息** 里发起 N 个 `Agent` (Task) 工具调用（一次一条 ID），这样它们并发运行不互相阻塞。

**进度行由主 skill 统一更新**——子 agent 不得碰 `tasks.md` 的"## 进度"段,只勾自己负责的任务的交付物 checkbox。所有 agent 返回后主 skill 重算并写一次进度。

每个 agent 的 prompt 模板（**把完整指令内联到 prompt 里,不要只写"按 fs-dev SKILL.md"**——子 agent 不继承本 skill）：

```
你扮演 fs-dev 子工程师,独立完成 {{T-xxx 或 BUG-xxx}} 的实施。

文档目录: {{docs_dir}}（把真实值填进去）

【Step 1 · 读上下文】
- {{T-xxx}}：Read `{{docs_dir}}/tasks.md` 找对应任务定义,再 Read 关联 FR / P / T / ADR 段落
- {{BUG-xxx}}：Read `{{docs_dir}}/bugs.md` 只读对应 BUG 条目（**不要读 qa.md 全文**） + 关联 FR / P / T
- Read `{{docs_dir}}/technical.md` 的目录结构与技术栈
- 前端相关：Read `{{docs_dir}}/prototype/README.md`,按字段中的 P-ID 查到文件路径后 Read 具体原型文件；样式规范看 `{{docs_dir}}/design.md`

【Step 2 · 对齐】
在简报里告诉主 skill 你准备怎么改（不用等确认,直接进 Step 3）。

【Step 3 · 实施】
严格按 technical.md 的目录 / 选型;不改范围外代码。前端任务遇 prototype 与 design.md 冲突时,视觉 / 交互 / 布局以 prototype 为准,色 / 字 / 间距 token 以 design.md 为准。

【Step 4 · 自检】
- 跑 `lint` / `typecheck` / 启动看页面（不跑 test、不创建测试文件）
- 任务：对照交付物 checkbox 逐条核对
- Bug：尝试复现步骤,观察现象是否消失

【Step 5 · 收尾 —— 按自检结果走】
- 任务 · 自检通过：在 tasks.md 勾自己这条任务的 **交付物 checkbox**；**不要** 改 `## 进度` 段（主 skill 最后统一重算）
- 任务 · 自检失败：**不勾** checkbox,简报里标 `❌ 失败` + 原因
- Bug · 现象消失：回写 bugs.md —— 状态 → `✅ 已修待重测`,填 `**修复**` 字段（文件 / 摘要 / 日期）
- Bug · 现象依然存在：**不要** 改为 `✅ 已修待重测`；保留 `🔧 修复中` 或回到 `❌ 待修`,`**修复**` 字段写"尝试了 X,未生效";简报标 `❌ 失败`
- Bug · 无法复现（缺真实环境）：改 `✅ 已修待重测`,`**修复**` 字段注明"未能本地复现,需 fs-qa 真实环境验证";简报标 `⚠️ 需重测`

【禁止】
- 动任务 / BUG 范围外的代码
- 动 tasks.md 的 `## 进度` 段（主 skill 职责）
- 动除 tasks.md / bugs.md 之外的上游文档（design.md、technical.md 等）
- 遇决策分歧（选型冲突、文档缺失、需求矛盾）自己拍板 —— **停下**,在简报里标 `⚠️ 需决策` 说明情况

【返回】100 字以内简报:做了什么 / 动了哪些文件 / 自检结果 / 标签（`✅ 完成` / `❌ 失败` / `⚠️ 需决策` / `⚠️ 需重测`）。
```

**汇总**：

所有 agent 返回后，主 fs-dev 依次:
1. **重算并更新** `tasks.md` 的 `## 进度` 段（避免并发写竞态）
2. 给用户一次性总结：每条的结果及下一步建议（`⚠️ 需决策` → 建议切回串行处理那一条；`❌ 失败` → 建议看子 agent 简报再决定）

### 风险与底线

- **写入竞态**：多个 agent 并发编辑同一文件会导致后写覆盖先写。并行前 **必须** 确认预估文件不重叠——拿不准就走串行
- **进度行集中更新**：子 agent 只勾 checkbox,`## 进度` 由主 skill 所有 agent 返回后重算,避免 race
- **状态推进由 agent 各自写回 tasks.md / bugs.md 主体**（打 checkbox、填 BUG 状态），主 skill 不代写这部分
- **默认保守**：用户没说"并行"就走串行。速度不是目标，正确才是
