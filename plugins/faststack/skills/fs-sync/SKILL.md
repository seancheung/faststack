# fs-sync — 文档变更同步

职责：**变更发生时，把它传播到所有受影响的上游文档，并记录变更理由与范围**。
不改代码、不做新拆解 —— 那是 `fs-dev` / `fs-tasks` 的事。

## 读取配置

读取 `.faststack.yml`，解析 `docs_dir` 和 `features`（未定义字段按 `false`）。

本 skill 涉及的文档集合：`idea` / `requirements` / `product` / `design` / `technical` / `tasks` / `qa` / `bugs`（以实际存在的为准）。

**feature 开关**：无。本 skill 不受 features 影响。

## 第一步：收集变更意图

用 `AskUserQuestion` 或直接问用户 3 件事：

1. **变更是什么**：一句话描述改了什么 / 要改成什么
2. **从哪一步发起**：开发中发现？产品评审改的？用户反馈后改的？
3. **已知影响的 ID**（可选）：知道就告诉我 FR-xxx / P-xxx / T-xxx，不知道我来找

## 第二步：分析影响范围

从 **源头** 向 **下游** 扫描所有现存文档，列出影响清单让用户确认。示例：

```
变更影响：
  requirements.md §3.2 · FR-010  描述改写
  product.md      §4.1 · P-003   新增一个步骤
  technical.md    §5  · /api/xx  参数变化
  tasks.md        T-105          已完成任务需重做
                  T-120          新增
  qa.md           TC-007         需重测
                  TC-030         新增
```

列清楚后必须让用户确认再进入第三步。漏扫或误伤都要就地更正。

## 第三步：交互式更新 —— 从上往下

按以下顺序逐份更新，每份让用户确认后再进入下一份。**必须从源头开始**，下游才有一致的基础。跳过当前项目中不存在的文档。

1. `idea.md`（如果动了 MVP / 边界 / 目标用户）
2. `requirements.md`
3. `product.md`
4. `design.md` / `prototype/`（如涉及 UI）
5. `technical.md`
6. `tasks.md`
7. `qa.md`（若存在，且受影响的 FR / P / T 有对应用例 → 更新用例 / 新增 / 标记重测）
8. `bugs.md`（若存在，检查每条活跃 BUG 的关联是否被变更触及 → 作废 / 改描述 / 追加备注；见下文特殊处理）

### 影响标注统一格式

所有受影响的条目都在对应位置追加 **一行** 影响标注，格式统一为：

```
> CR-xxx：{一句话影响}
```

下游 skill（fs-dev / fs-qa）检测到该格式就知道这条需要关注。不要用"需重做"/"需重测"/"受影响"等多套措辞——统一由这一行的 `{影响}` 文字表达。

### `tasks.md` 的特殊处理

- **已完成且受影响** 的任务：在任务下面加 `> CR-xxx：需重做，{具体影响}` —— **不要擅自取消勾选**，由 `fs-dev` 判断怎么处理
- **进行中** 的任务：更新范围描述，同样追加 `> CR-xxx：{影响}`
- **新任务**：追加到合适的里程碑，T-xxx 编号续号

### `qa.md` 的特殊处理

- **已通过**（结果 ✅）且受影响的用例：在用例下面加 `> CR-xxx：需重测，{具体影响}` —— **不要擅自改成 ⏳**，由 `fs-qa` 执行时决定
- **未执行** / **已失败** 的用例：直接更新步骤 / 预期，保留现有结果
- **新用例**：追加，TC-xxx 编号续号

### `bugs.md` 的特殊处理

- **活跃 BUG 受影响**（BUG 的关联 FR / P / T 被变更触及）：
  - 变更后 BUG 依然成立 → 追加 `> CR-xxx：复现步骤 / 现象需重核，{具体影响}`，不改状态
  - 变更后 BUG 已失效（如需求被删） → 状态改 `⏭️ 作废`，追加 `> CR-xxx：作废，{原因}`，不挪到存档
- **已修待重测 的 BUG**：追加 `> CR-xxx：重测时注意新行为，{具体差异}` —— 不回退状态
- **存档 BUG**：一般不动；仅当变更明显把老问题重新引入时才复活一条新 BUG（不改老条目）

## 第四步：记录变更日志

在 `{docs_dir}/changelog.md` 追加条目（不存在就新建）：

```markdown
# 变更日志

## CR-001 · 2026-04-19
**发起方**：开发阶段（`fs-dev`，任务 T-105）
**变更**：登录方式从邮箱改为手机号 + 邮箱双通道
**理由**：用户调研发现目标群体主要用手机
**影响**：FR-010, P-003, /api/auth/login, T-105（重做）, +T-120
**文档更新**：requirements.md §3.2, product.md §4.1, technical.md §5, tasks.md M2
```

- CR 编号 **递增**（CR-001, CR-002, ...）
- 日期用 **绝对日期**（2026-04-19），不要写"今天"
- "发起方"要写出具体上下文，便于后续回查

## 完成后

向用户汇报：
- 改了哪几份文档 + 哪些任务受影响
- 新增的 CR 编号
- 建议下一步：回 `fs-dev` 继续 / 重新 `fs-tasks`（如任务重排较大）/ 回对应 skill 深度修订

## 不要做的事

- **不要直接改代码实现** —— 那是 `fs-dev`
- **不要跳过 changelog 记录** —— 变更追溯是 sync 的核心价值
- **不要把"补充未写过的细节"当变更处理** —— 提示用户直接去对应 skill 补写即可
- **不要把多个不相关的变更混合在一个 CR 里** —— 建议分次 sync，每次一个主题
- **不要反向传播**（下游覆盖上游后停下）—— 必须一路走完到 `tasks.md`，不然会出现文档之间互相矛盾
