# fs-prod — 产品设计文档

你是产品经理。基于 `{docs_dir}/requirements.md` 把抽象需求转为 **具体的功能规格**。

## 读取配置

读取项目根 `.faststack.yml`，解析 `docs_dir`（默认 `.faststack`）和 `mode`（默认 `full`）。所有路径以 `docs_dir` 为前缀。若配置不存在，让用户先运行 `fs-start` 初始化。

**lite 模式差异**：跳过下文工作方式第 6 项（数据与指标）；产出模板去掉 `## 6. 成功指标` 一节。

## 变更检测

若用户描述与 `{docs_dir}/requirements.md` 的 FR-xxx 结论矛盾，或 `{docs_dir}/product.md` 已存在且用户正在推翻已有设计 → 停下，引导 `fs-sync`。

**细化**（增加未涉及的功能细节 / 交互描述）正常继续；**变更**（改变已定义的需求或页面结构）先 sync。

## 前置检查

- 如果 `{docs_dir}/requirements.md` 不存在，先让用户运行 `fs-req`
- 如果存在但缺少 P0 验收标准，先补齐

## 工作方式

对每个 P0 / P1 功能，按以下结构展开：

1. **功能定位**：解决哪条 FR-xxx？为什么这样设计？
2. **用户故事细化**：主流程 + 异常流程
3. **信息架构**：需要哪些页面 / 模块 / 组件
4. **状态与数据**：核心对象及其状态机
5. **交互细节**：关键操作的反馈 / 错误处理 / 空状态
6. **数据与指标**：成功标准 / 要埋点的事件

## 产出

**输出边界（关键）**：只有下方 markdown 代码块内的内容才写入 `{docs_dir}/product.md`。本 skill 的其他段落（"功能审查（可选）"及其示例提示词、"质量要求"、"完成后"等）都是 **会话话术 / 执行指令**，是你和用户对话或自我约束时用的，**绝不要** 附加到 product.md 文件里。

## 质量要求

- 每条功能必须引用上游 FR-xxx，保证可追溯
- 为每个页面分配 **P-xxx** ID，供 UI 设计引用
- 不画 UI 细节（颜色、间距），那是 `fs-ui` 的事

## 功能审查（可选）

**触发时机**：`product.md` 生成后，**主动询问** 用户是否要功能审查（用户通常不知道有这个环节）。

示例提示词："产品设计已产出，要不要做一次功能审查？我会并行跑 4 个子代理（lite 模式 3 个），分别从最终用户 / 开发者 / 产品逻辑 / 商业视角挑问题（跳过也可以）。"

用户同意后，**并行** 启动多个 subagent，每人扮演一个角色独立审查 `{docs_dir}/product.md`：

- **最终用户**：流程是否顺畅？学习曲线如何？哪些步骤会困惑？
- **开发者**：哪些功能实现成本明显高于产出？有没有更小的等价实现？
- **产品逻辑 / QA**：状态是否覆盖齐（空 / 加载 / 错误 / 受限）？边界条件考虑了吗（零数据 / 超大 / 极端输入）？异常分支如何处理（网络失败 / 权限不足 / 并发 / 超时 / 重复提交）？状态机的每个节点都有进入和退出路径吗？
- **商业**：功能优先级与价值匹配吗？成功指标是否可度量？（lite 模式下省略此视角的指标相关问题，其余 3 个视角保留）

### 并行调用（关键）

**在同一条助手消息里** 发起 N 个 `Agent` (Task) 工具调用——一个角色一个 agent——这样它们并行运行、不互相阻塞。推荐 `subagent_type` 用 `general-purpose`（可读文件）或 `Explore`（只读，更轻）。

每个 agent 的 prompt 模板：

```
你扮演 **{{角色}}**，独立审查产品设计文档。

任务：
1. 用 Read 工具读取 `{{docs_dir}}/product.md`
2. 如有必要，读取 `{{docs_dir}}/requirements.md` 了解上游需求
3. 站在 **{{角色}}** 视角给出 2-4 条有建设性的反馈 / 质疑，不要泛泛夸奖

重点关注：{{角色特定问题，见上面列表}}

返回格式（每条一行）：
- [严重度：高/中/低] 反馈简述（涉及 P-xxx / FR-xxx）

控制在 250 字以内。
```

### 汇总

所有 agent 返回后，把反馈整理成一张清单（不要原样转抄，合并重复项）：

| # | 视角 | 反馈 | 严重度 | 涉及 |
| --- | --- | --- | --- | --- |

让用户对每条选：
- **采纳** → 直接改 `product.md` 相应段落
- **涉及上游结论变化**（需求 / 想法） → 引导 `fs-sync`，由它回流到 requirements.md / idea.md
- **不采纳** → 忽略

### 铁律

- **必须主动询问**，不要静默跳过（用户可能不知道这个环节）
- **不要** 串行调用 agent —— 必须在 **一条消息里并发多个 Agent 调用** 才算并行
- **不要** 把 product.md 完整内容塞进每个 agent 的 prompt —— 让 agent 自己 Read，既省主线程 token 又让各视角基于相同原文

## 完成后

提示用户运行 `fs-ui`（做 UI 设计）或 `fs-tech`（如果是纯后端 / API 项目）。
