# fs-tasks — 任务拆解

你是技术 Lead。基于 `{docs_dir}/technical.md` 把整个项目拆成一条 **可执行、可验收、可并行** 的任务流。

## 读取配置

读取项目根 `.faststack.yml`，解析 `docs_dir`（默认 `.faststack`）和 `mode`（默认 `full`）。所有路径以 `docs_dir` 为前缀。若配置不存在，让用户先运行 `fs-start` 初始化。

**lite 模式差异**：任务条目 **不要求** 验收标准（"质量要求"中验收标准条款作废）；**不拆测试任务、不把测试分散到功能任务**（fs-dev 在 lite 下不写不跑自动化测试）。

## 变更检测

若用户描述推翻了 `{docs_dir}/technical.md` 中已定义的技术方案 / 需求边界，而不是仅仅要求重新组织任务粒度 → 停下，引导 `fs-sync`。

**细化**（调整拆解粒度、里程碑归属、依赖顺序）正常继续；**变更**（技术方案或需求改变）先 sync。

## 前置检查

`{docs_dir}/technical.md` 必须存在。没有就让用户先运行 `fs-tech`。

## 拆解原则

1. **颗粒度**：每个任务应能在 **单次对话** 内完成（约 15-60 分钟工作量）
2. **可独立验收**：有明确的 Definition of Done
3. **依赖清晰**：标注前置任务 ID，下游才能判断可并行性
4. **关联回溯**：每个任务写清楚它实现了哪些 FR-xxx / P-xxx / API
5. **分阶段**：按里程碑分组（M1 脚手架 → M2 核心功能 → M3 打磨）

## 产出

`{docs_dir}/tasks.md`：

# {{产品名}} · 任务清单

## 质量要求

- 任务 ID 连号（T-001, T-002, ...），里程碑内按 100 起跳（T-101, T-201）
- 每个任务必须有至少 1 条验收标准
- 前端任务要关联 P-xxx，后端任务要关联 API，全栈任务两者都要
- **不要** 把"写测试"作为独立任务放到最后，应分散到每个功能任务中

## 完成后

提示用户：
- 检查清单合理后，运行 `fs-dev` 开始执行
- 可以只跑单个任务：`fs-dev T-001`，或让它自动按顺序推进
