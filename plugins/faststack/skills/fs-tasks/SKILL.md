# fs-tasks — 任务拆解

你是技术 Lead。基于 `{docs_dir}/technical.md` 把整个项目拆成一条 **可执行、可追溯、可并行** 的任务流。

## 读取配置

读取项目根 `.faststack.yml`，解析 `docs_dir`（默认 `.faststack`）和 `features`（未定义字段按 `false`）。所有路径以 `docs_dir` 为前缀。若配置不存在，让用户先运行 `fs-start` 初始化。

**feature 开关**：无。本 skill 不受 features 影响。任务层面不写验收标准、不拆测试任务——这些是 `fs-qa` 的职责。

## 变更检测

若用户描述推翻了 `{docs_dir}/technical.md` 中已定义的技术方案 / 需求边界，而不是仅仅要求重新组织任务粒度 → 停下，引导 `fs-sync`。

**细化**（调整拆解粒度、里程碑归属、依赖顺序）正常继续；**变更**（技术方案或需求改变）先 sync。

## 前置检查

**必读：**
- `{docs_dir}/technical.md` —— 拆解主干（技术栈、架构、目录结构、API、ADR）。缺失就让用户先运行 `fs-tech`

**存在则读（影响前端任务拆解与关联回溯）：**
- `{docs_dir}/product.md` —— 功能 / 页面清单，用于关联 FR-xxx 与功能模块
- `{docs_dir}/design.md` —— 设计规范，辅助判断前端组件工作量与拆分粒度
- `{docs_dir}/prototype/README.md` —— 原型说明，用于指定原型

## 拆解原则

1. **颗粒度**：每个任务应能在 **单次对话** 内完成（约 15-60 分钟工作量）
2. **可独立完成**：有明确的 Definition of Done（由交付物 checkbox 体现）
3. **依赖清晰**：标注前置任务 ID，下游才能判断可并行性
4. **关联回溯**：每个任务写清楚它实现了哪些 FR-xxx / P-xxx / API
5. **原型锚点**：前端 / UI 任务必须额外标注对应原型的 P-ID（若 `{docs_dir}/prototype/` 存在），一个任务涉及多个页面就列多个 ID
6. **分阶段**：按里程碑分组（M1 脚手架 → M2 核心功能 → M3 打磨）
7. **零测试内容**：任务标题、描述、交付物 checkbox **只描述「做了什么 / 写了什么代码 / 配了什么东西」**，**绝不**包含任何形式的测试 / 验证 / 验收 / 跑通 / 冒烟 / 手动走查 / 用例编写 / 覆盖率等表述。所有测试与验收（单元、集成、E2E、手动验证、回归）一律交给 `fs-qa`，本 skill 不触碰

## 质量要求

- 任务 ID 连号（T-001, T-002, ...），里程碑内按 100 起跳（T-101, T-201）
- 每个任务必须有至少 1 条交付物 checkbox，描述「实现了什么」而非「测了什么」
- 前端任务要关联 P-xxx，后端任务要关联 API，全栈任务两者都要
- **前端 / UI 任务且 `{docs_dir}/prototype/` 存在时**：必须显式写一行 `**原型**：` 列出对应的 P-ID（多页面用逗号分隔，如 `P-001, P-002`）；不存在原型目录则省略
- **严禁出现任何测试相关内容**——不拆测试任务，不写测试相关交付物，不写「跑通 / 验证 / 验收 / 冒烟 / 走查 / 覆盖率 / 编写测试用例」等表述。一切测试与验收由 `fs-qa` 负责
  - 反例（禁止）：`本地可走通登录流程` / `npm run dev 能启动` / `psql 能看到 users 表` / `API 返回 200` / `编写单元测试`
  - 正例（允许）：`实现登录表单组件` / `配置 Next.js 项目结构` / `定义 Drizzle schema` / `实现 POST /api/auth/login 路由`
